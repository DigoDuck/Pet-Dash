# PetDash Fase 1 — API (DRF) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Expor os 8 models da Fase 0 via API REST autenticada (JWT), com CRUD de cadastros, atendimentos com pagamentos aninhados e validação financeira, venda de pacote, e um endpoint de dashboard com KPIs derivados — em 4 PRs revisáveis (PR 8 de deploy fica para esboço posterior).

**Architecture:** DRF `ModelViewSet` + routers sobre o app único `core`. Auth por `djangorestframework-simplejwt` (todas as rotas exigem token, exceto healthcheck e obtain/refresh). Filtros por `django-filter` + `SearchFilter`. A regra financeira **nunca** é reimplementada na view: serializers validam invariantes e o dashboard reusa `core.services` (faturamento) + novas agregações em `core.services`. Serializers e views ficam em arquivos próprios (`core/serializers.py`, `core/views.py`, `core/urls.py`).

**Tech Stack:** Django 5.2 · DRF 3.17 · djangorestframework-simplejwt · django-cors-headers · django-filter · pytest-django · factory-boy · ruff.

## Global Constraints

- Todo código backend vive em `backend/`. Comando Python nas tasks = `backend/.venv/Scripts/python.exe` (Windows).
- Testes sempre contra PostgreSQL via `DATABASE_URL`. SQLite é proibido.
- Commits em inglês, conventional (`feat:`, `test:`, `chore:`), **sem trailer de coautoria**. Comentários/docstrings podem ser PT-BR.
- Fluxo por PR: branch a partir de `main` → commits → push → abrir PR → **parar, aguardar revisão + CI verde + merge** antes do PR seguinte.
- `ruff check .` limpo e `pytest` verde são obrigatórios para fechar qualquer PR.
- Invariantes da spec que a API precisa respeitar (não flexibilizar):
  - `sum(Pagamento.valor) == Atendimento.valor` para avulsos (`pacote_id IS NULL`), validado no serializer.
  - Consumo de pacote (`pacote_id` preenchido) não valida soma de pagamentos (o pagamento já foi na venda do pacote) e não entra no faturamento.
  - Saldo do pacote = `qtd_total - COUNT(atendimentos WHERE status != 'Cancelado')`; não marcar atendimento em pacote sem saldo.
  - `UNIQUE(pet, competencia)`; `competencia` normalizada para o dia 1 (já no model).
  - Soft-delete de Tutor/Pet via `ativo` (o `destroy` da API seta `ativo=False`, não deleta).
  - Financeiro derivado por query, nunca materializado.

```python
class PagamentoSerializer(serializers.ModelSerializer):
    class Meta:
        model = models.Pagamento
        fields = ["id", "metodo", "valor"]


class AtendimentoSerializer(serializers.ModelSerializer):
    pagamentos = PagamentoSerializer(many=True, required=False)

    class Meta:
        model = models.Atendimento
        fields = [
            "id", "pet", "servico", "pacote", "data", "horario", "valor",
            "transporte", "transporte_valor", "status", "pagamentos",
        ]

    def create(self, validated_data):
        pagamentos = validated_data.pop("pagamentos", [])
        atendimento = models.Atendimento.objects.create(**validated_data)
        for pag in pagamentos:
            models.Pagamento.objects.create(atendimento=atendimento, **pag)
        return atendimento

    def update(self, instance, validated_data):
        pagamentos = validated_data.pop("pagamentos", None)
        for attr, value in validated_data.items():
            setattr(instance, attr, value)
        instance.save()
        if pagamentos is not None:
            instance.pagamentos.all().delete()
            for pag in pagamentos:
                models.Pagamento.objects.create(atendimento=instance, **pag)
        return instance
```
